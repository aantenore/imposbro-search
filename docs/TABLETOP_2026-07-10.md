## Scenario 1 — PostgreSQL corruption and clean restore

### Inject and impact

Assume the authoritative control-plane database returns an invalid state
digest or audit-head invariant after a storage fault. Admin mutations, routing
transitions, event publication, and worker checkpoint correctness are now
untrusted. Serving a previously loaded read snapshot is permitted only while
its identity and authorization policy remain demonstrably intact; otherwise
the affected replica must leave readiness.

### Decisions

1. Declare SEV-1, freeze control-plane mutations, routing rollouts, retention,
   and release activity, and preserve database/storage evidence. Do not edit a
   revision, audit row, outbox marker, or checkpoint to make health green.
2. Record the last known-good release digests, Alembic revision, control-plane
   revision/digest, audit head, durable business marker, Kafka/outbox position,
   and incident start time. Bound or pause ingest if PostgreSQL capacity or
   event durability is no longer provable.
3. Select an independently verified age-encrypted backup. Restore only into a
   new, isolated, empty target through the guarded transactional restore path;
   never overwrite the suspected source.
4. Verify schema head, state digest, non-empty audit chain, outbox/checkpoint
   invariants, source/restore logical fingerprints, and known deletion
   suppression before connecting application replicas.
5. Calculate RPO from the latest durable business marker and RTO through
   authenticated application validation, not from archive creation or schema
   restore alone. Cut over only with platform, service, data/security, and
   change approval. Preserve the former database for the approved forensic
   window, then destroy it under policy.

### Repository evidence and conclusion

- [ADR 0001](adr/0001-transactional-control-plane-store.md) defines the
  authority, transaction, migration, and no-silent-downgrade boundary.
- [The backup/restore runbook](runbooks/postgres-backup-restore.md),
  `scripts/ops/backup-control-plane-postgres.sh`, and
  `scripts/ops/restore-control-plane-postgres.sh` implement fail-closed local
  mechanics.
- `evidence/dr-live-postgres-latest.json` records a passing disposable local
  exercise, but its provenance identifies a dirty working tree and explicitly
  denies production certification.

The decision path is **viable for a repository-owned isolated restore**.
Scheduled backup execution, immutable off-site custody, key recovery, managed
failover, application cutover, production RPO/RTO, and deletion-after-restore
proof remain deployment-owned and open.

## Scenario 2 — Kafka lag growth and unresolved DLQ

### Inject and impact

Assume consumer lag rises continuously while failed indexing records enter the
DLQ. Search data becomes stale, an older event may be retried after a tombstone,
and an in-progress routing cutover can no longer prove parity or convergence.

### Decisions

1. Stop routing cutover/completion and unrelated schema or release changes.
   Record topic, partition, group, committed/end offsets, first affected event,
   outbox age, worker readiness, DLQ growth, routing revision, and rollout ID.
2. Keep accepted events in the durable outbox/Kafka path. Rate-limit or pause
   new ingest before PostgreSQL or Kafka exhausts capacity; never purge an
   outbox, reset a consumer group, advance an offset, or widen retry budgets to
   hide the incident.
3. Restore broker, ACL, TLS/SASL, network, quota, disk, worker, or downstream
   Typesense health through the owning provider procedure. Re-establish
   monotonic drain before increasing concurrency.
4. Inspect exactly one next-unresolved DLQ record using the repository resolver.
   Validate envelope, key, tenant/collection, sequence, route, and root cause.
   Replay only after producer acknowledge and downstream checkpoint proof; a
   non-replay disposition needs data/service approval and security approval for
   authorization cases.
5. Resume cutover only after outbox and primary consumer lag are within budget,
   resolver-group lag is exactly zero, no new DLQ item appears during the
   stabilization interval, target checkpoints agree, and parity is recomputed.

### Repository evidence and conclusion

- [ADR 0004](adr/0004-at-least-once-indexing-consistency.md) defines replay,
  fencing, tombstone, and offset-commit semantics.
- [The outbox/DLQ runbook](runbooks/outbox-lag.md) and
  `scripts/ops/dlq_resolver.py` define the bounded single-record recovery path.
- Routing state-machine tests close cutover when Kafka or DLQ evidence is
  outside budget. The current live composite evidence is not passing and
  therefore does not prove this path on the release candidate.

The recovery decisions preserve data, but direct consumer-lag, unresolved-DLQ,
and rollout-failure paging coverage was incomplete at review time. That finding
is **remediation in progress** and must be closed with rule tests plus live alert
delivery evidence.

## Scenario 3 — Compromised credential and forced rotation

### Inject and impact

Assume a credential used by the Admin UI, Query API, worker, or a Typesense
cluster is disclosed and unauthorized use is suspected. Treat possible
cross-tenant access, control-plane mutation, document access, and backup
exposure as SEV-1 until scope is bounded.

### Decisions

1. Freeze risky mutations/releases, preserve redacted authentication/audit
   evidence, identify the principal, scope, provider versions, affected
   clusters/tenants, first/last use, sessions, backups, and secret-manager
   access. Never place the disclosed value in a ticket, command, log, or chat.
2. Issue a new least-privilege provider credential and update the authoritative
   secret manager. For a known compromise, do not restore the old value as a
   rollback; minimize or remove overlap under the provider's emergency
   rotation procedure.
3. Change only the declared non-secret rotation version and perform a bounded
   rollout without rebuilding images. Verify strict readiness and authenticated
   admin, tenant search, ingest/worker, audit, and trace journeys on the new
   identity.
4. Revoke the old credential and prove that old-version use fails. Rotate the
   Admin UI session secret or revoke IdP sessions/tokens as applicable. Search
   for unauthorized actions by actor/request ID and treat each affected tenant
   under the incident and notification policy.
5. Keep backups that contain the old secret protected until approved expiry or
   crypto-erasure. A restore must rotate or suppress the compromised value
   before any traffic is served.

### Repository evidence and conclusion

- [ADR 0005](adr/0005-enterprise-security-boundaries.md),
  [the threat model](THREAT_MODEL.md), and
  [the rotation runbook](runbooks/credential-rotation.md) define fail-closed
  startup, scoped workload injection, and provider-specific rotation order.
- Helm exposes an external-secret delivery and rollout mechanism without image
  rebuilds; local tests do not prove provider issuance, revocation, or IdP
  behavior.

The mechanism is usable for environment-injected credentials. Typesense keys
were still persisted in control-plane state and protected backups at review
time, matching threat-model risk T-03; migration to secret references or an
equivalent KMS-backed design is **remediation in progress**. Live old-key
rejection, session revocation, key-custody, and backup handling remain open
deployment evidence.

## Scenario 4 — Routing cutover failure and rollback

### Inject and impact

Assume the candidate Typesense target degrades after cutover but before the
rollback window ends, or a post-cutover probe exposes stale/missing documents.
New reads prefer the candidate while delete/write coverage still spans both
policies. Completing the rollout would make recovery unsafe.

### Decisions

1. Freeze the rollout, capture its global revision, local version, phase,
   active/candidate policies, parity/barrier/checkpoint, lag, DLQ, target health,
   request IDs, and failure time. Do not edit routing JSON or operator-submit
   machine evidence.
2. Stop completion and new rollout work for the collection. Keep broad
   read/delete/write coverage supplied by the persisted lifecycle; do not purge
   candidate or source documents during diagnosis.
3. If rollback is required, refetch authoritative state and submit the legal
   CAS transition to `rolling_back`, then validate active-policy search,
   ingest, delete, checkpoints, outbox drain, and configuration convergence on
   every replica before transitioning to `rolled_back`.
4. A `409` means the operator view is stale: refetch and re-evaluate instead of
   blind retry. A provider or persistence failure leaves the durable desired
   state for reconciliation and remains an incident.
5. Reconcile or quarantine candidate copies only after rollback validation.
   Preserve the immutable rollout/audit record and open a new rollout for any
   later attempt; never reuse or decrement the failed workflow.

### Repository evidence and conclusion

- [ADR 0003](adr/0003-safe-routing-migrations.md) and state-machine tests define
  union coverage, CAS, barriers, parity, cutover, and rollback.
- [The routing failure runbook](runbooks/routing-rollout-failure.md) records the
  operator procedure created from this exercise.
- The Admin UI exposes revision/version conflicts, gate state, audit history,
  confirmation, and rollback transitions. The current live two-cluster
  cutover/rollback artifact is not passing, so runtime recovery remains open.

The state-machine decision path is coherent. Direct rollout failure/deadline
alerts and exact-commit two-cluster rollback evidence are **remediation in
progress**; no release claim is made.
